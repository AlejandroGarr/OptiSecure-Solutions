1.1. Motivación
───────────────

El sector de la seguridad doméstica y empresarial se encuentra en un proceso
de transformación digital acelerado. Las empresas de este ámbito necesitan
herramientas que les permitan gestionar de forma centralizada su parque de
cámaras de vigilancia, atender las solicitudes de contratación de nuevos
dispositivos, resolver incidencias técnicas y, al mismo tiempo, ofrecer a sus
clientes un canal de autoservicio donde puedan supervisar sus instalaciones
de forma autónoma.

Tradicionalmente, la construcción de un sistema con estas características
requiere abordar múltiples disciplinas de ingeniería de software de forma
simultánea: diseño de bases de datos, desarrollo de un servidor backend,
implementación de una interfaz web, configuración de infraestructura en la
nube, sistemas de autenticación y control de acceso, y despliegue continuo.
Este enfoque, si bien ofrece plena libertad arquitectónica, conlleva plazos
de desarrollo prolongados y costes de mantenimiento elevados.

La plataforma Salesforce ofrece una alternativa que permite concentrar el
esfuerzo de desarrollo en la lógica de negocio, al proporcionar de forma
nativa un motor de base de datos, un entorno de ejecución server-side, un
framework de componentes web, un portal de clientes con autenticación
integrada y un sistema de permisos granular. Esta premisa constituye la
motivación principal de este Trabajo Fin de Grado: demostrar que es posible
construir una aplicación de dominio específico completa sobre Salesforce,
reduciendo drásticamente el tiempo de desarrollo sin sacrificar funcionalidad
ni calidad del producto.

Adicionalmente, el proyecto integra capacidades de inteligencia artificial
mediante la API de Google Gemini y un sistema de streaming de vídeo local
mediante Flask, Ngrok y OpenCV, ampliando el alcance funcional de la
plataforma más allá de las capacidades nativas de Salesforce.


1.2. Objetivos
──────────────

El presente Trabajo Fin de Grado persigue los siguientes objetivos:

  1. Diseñar e implementar una aplicación completa de gestión de seguridad
     sobre la plataforma Salesforce, abarcando desde el modelo de datos hasta
     el portal de clientes.

  2. Demostrar que Salesforce es viable como plataforma de desarrollo de
     aplicaciones de dominio específico, más allá de su uso convencional
     como CRM.

  3. Integrar inteligencia artificial externa (Google Gemini) con Salesforce
     para el procesamiento de comandos de voz en lenguaje natural.

  4. Desarrollar una interfaz de usuario moderna y funcional mediante
     Lightning Web Components, aprovechando estándares web actuales.

  5. Implementar un sistema de streaming de vídeo en tiempo real que permita
     visualizar cámaras de seguridad locales desde el portal web de
     Salesforce mediante Flask, OpenCV y Ngrok.

  6. Cubrir el código backend con pruebas unitarias automatizadas,
     incluyendo simulaciones de llamadas a APIs externas.

  7. Aplicar un modelo de seguridad robusto basado en perfiles, permisos y
     verificaciones programáticas server-side.

              CAPÍTULO 2. ESTADO DEL ARTE

================================================================================

───────────────────────────────────────────

El mercado de las plataformas de Customer Relationship Management ha
experimentado un crecimiento sostenido durante la última década. Según los
informes sectoriales más recientes, Salesforce mantiene la posición de
liderazgo con una cuota de mercado superior al 23 %, seguida de Microsoft
Dynamics 365, SAP CRM y Oracle CX Cloud.

La evolución de estas plataformas ha trascendido la gestión comercial pura
para convertirse en entornos de desarrollo de aplicaciones empresariales
completas. El paradigma low-code/no-code permite reducir los ciclos de
desarrollo al proporcionar componentes reutilizables, motores de
automatización visual y modelos de datos configurables. Sin embargo, para
aplicaciones que requieren lógica de negocio compleja, integraciones con
servicios externos o interfaces de usuario personalizadas, el desarrollo
programático sigue siendo imprescindible.

La principal ventaja competitiva de Salesforce como plataforma de desarrollo
reside en que proporciona, de forma nativa, un motor de base de datos
multi-tenant, un sistema de permisos granular, un framework de componentes
web (LWC), un portal de clientes (Experience Cloud), una API REST/SOAP
automática y un entorno de ejecución server-side (Apex). Esto permite al
desarrollador concentrarse exclusivamente en la lógica de negocio específica
de su dominio.


2.1 Salesforce como plataforma de desarrollo
──────────────────────────────────────────────

Salesforce ofrece un ecosistema de desarrollo que cubre las necesidades
habituales de una aplicación empresarial:

  - Base de datos incluida: Los objetos personalizados (Custom Objects)
    funcionan como tablas con campos, relaciones, validaciones y reglas de
    cálculo. No se requiere instalación ni configuración de ningún motor
    de base de datos externo.

  - Backend incluido: Apex es el lenguaje de programación server-side de
    Salesforce, similar a Java en su sintaxis. Permite escribir
    controladores, ejecutar consultas SOQL, realizar llamadas HTTP a APIs
    externas y gestionar transacciones atómicas.

  - Portal de clientes incluido: Experience Cloud proporciona un sitio web
    listo para el registro, la autenticación y el acceso seguro de los
    clientes a sus datos. Implementar una funcionalidad equivalente con
    React y Auth0 requiere un esfuerzo significativamente mayor.

  - Seguridad integrada: Perfiles, Permission Sets, Sharing Rules y
    Field-Level Security conforman un modelo de seguridad declarativo que
    se complementa con las directivas with sharing / without sharing en Apex.

  - Despliegue y versionado: Salesforce CLI (sf) permite la extracción y
    el despliegue de código y metadatos. El proyecto se versiona con Git
    siguiendo las prácticas estándar de la industria.

  - Integraciones: Apex puede realizar llamadas HTTP a APIs externas y
    Salesforce expone automáticamente sus datos mediante API REST/SOAP.


2.3. Seguridad doméstica inteligente
─────────────────────────────────────

El mercado de la seguridad doméstica inteligente se encuentra en plena
expansión, impulsado por la adopción masiva de dispositivos IoT (cámaras IP,
sensores de movimiento, cerraduras inteligentes) y por la creciente demanda
de los usuarios de controlar y supervisar sus hogares de forma remota.

Las soluciones existentes en el mercado (Ring, Arlo, Nest, Ajax Systems) se
centran en el hardware y proporcionan aplicaciones móviles propias para la
gestión del usuario final. Sin embargo, las empresas de seguridad que
comercializan e instalan estos dispositivos carecen, en muchos casos, de
herramientas centralizadas para:

  - Gestionar el inventario de cámaras desplegadas en múltiples clientes.
  - Tramitar de forma ordenada las solicitudes de contratación de nuevos
    dispositivos.
  - Ofrecer soporte técnico con trazabilidad y seguimiento de casos.
  - Proporcionar a cada cliente un acceso individualizado y seguro a su
    equipamiento.

OptiSecure Solutions aborda esta carencia ofreciendo una plataforma
centralizada que cubre tanto las necesidades operativas de la empresa de
seguridad como las necesidades de autoservicio de sus clientes.


2.4. Inteligencia Artificial aplicada a servicios empresariales
───────────────────────────────────────────────────────────────

La integración de modelos de lenguaje de gran escala (LLM) en aplicaciones
empresariales ha dejado de ser una tendencia emergente para convertirse en
una práctica habitual. Modelos como GPT-4, Claude o Gemini permiten
incorporar capacidades de procesamiento de lenguaje natural en flujos de
trabajo existentes sin necesidad de entrenar modelos propios.

En el contexto de OptiSecure Solutions, la IA se emplea para interpretar
comandos de voz de los clientes y convertirlos en acciones concretas dentro
de Salesforce (apertura de casos de soporte, consulta de estado de cámaras).
El modelo utilizado es Google Gemini 2.0 Flash, seleccionado por su velocidad
de respuesta, su capacidad de procesamiento en español y su disponibilidad
gratuita dentro del tier de uso personal.


2.5. Streaming de vídeo local en aplicaciones cloud
────────────────────────────────────────────────────

Uno de los desafíos técnicos del proyecto consiste en permitir la
visualización de cámaras de seguridad locales dentro de un portal web alojado en la nube pública
(Salesforce Experience Cloud).

Las soluciones de streaming convencionales (WebRTC, RTSP) requieren una
infraestructura de servidores de relay o TURN servers que encarecen y
complican la arquitectura. Para este proyecto se ha optado por una solución
ligera basada en tres componentes:

  - Un servidor de streaming local implementado en Python con Flask y
    OpenCV, que captura los fotogramas de la cámara y los emite como flujo
    MJPEG a través de una URL HTTP local.

  - Un túnel seguro proporcionado por Ngrok, que expone la URL local en
    una URL HTTPS pública accesible desde cualquier punto de Internet.

  - Un componente LWC en Salesforce que consume la URL pública de Ngrok
    mediante una etiqueta <img> con el flujo MJPEG, mostrando el vídeo
    en tiempo real dentro del portal del cliente.

Esta arquitectura permite demostrar el concepto de streaming de cámaras reales
sin requerir infraestructura de servidores intermedios, y resulta extensible
a escenarios de producción mediante sustitución de Ngrok por una solución de
tunneling empresarial o por streaming WebRTC directo.

================================================================================


3.1. Planificación y fases del desarrollo
─────────────────────────────────────────

El desarrollo de OptiSecure Solutions se ha estructurado en las siguientes
fases:

  Fase 1 — Análisis y diseño:
    Definición del modelo de negocio, identificación de requisitos
    funcionales y no funcionales, diseño del modelo de datos (objetos
    personalizados Camara__c y Solicitud_Camara__c) y diseño de la
    arquitectura por capas.

  Fase 2 — Configuración del entorno:
    Creación del proyecto SFDX, configuración del Sandbox, definición de
    perfiles y permisos, creación de la estructura de directorios y
    configuración de herramientas de desarrollo (VS Code, Salesforce CLI,
    Jest, ESLint).

  Fase 3 — Desarrollo del backend (Apex):
    Implementación de los controladores Apex (8 clases funcionales y 4
    clases de test), incluyendo la integración con la API de Google Gemini.

  Fase 4 — Desarrollo del frontend (LWC):
    Construcción de los 10 componentes Lightning Web Components funcionales,
    con sus respectivos archivos HTML, JavaScript, CSS y metadatos.

  Fase 5 — Configuración del portal:
    Despliegue y configuración del sitio Experience Cloud (LWR), incluyendo
    branding, perfiles de acceso, componentes expuestos y página de login
    personalizada.

  Fase 6 — Integración de streaming de vídeo:
    Desarrollo del servidor Flask con OpenCV para captura y emisión de
    vídeo local, configuración de Ngrok como túnel público e integración
    con el componente LWC del mapa de seguridad.

  Fase 7 — Pruebas y documentación:
    Ejecución de pruebas unitarias Apex, validación funcional del portal
    y redacción de esta documentación.


3.2. Metodología de desarrollo aplicada
───────────────────────────────────────

El proyecto se ha desarrollado siguiendo un enfoque iterativo-incremental,
realizando entregas funcionales al final de cada fase. Cada iteración ha
incluido un ciclo completo de diseño, implementación, prueba y validación.

Se ha utilizado Git como sistema de control de versiones, con un
repositorio único que contiene tanto el código fuente de Salesforce (SFDX)
como los scripts de Python para el servidor de streaming.


3.3. Herramientas de gestión
────────────────────────────

  - Visual Studio Code: Editor principal de desarrollo con Salesforce
    Extension Pack.
  - Salesforce CLI (sf): Herramienta de línea de comandos para despliegue

  - Git / GitHub: Control de versiones y alojamiento del código fuente.
  - Jest: Framework de testing para componentes LWC.
  - ESLint: Herramienta de análisis estático de código JavaScript.

              CAPÍTULO 4. TECNOLOGÍAS UTILIZADAS

================================================================================


4.1. Salesforce Platform
────────────────────────

Salesforce es una plataforma cloud de desarrollo de aplicaciones empresariales
que proporciona un entorno multi-tenant con base de datos integrada, motor de
ejecución server-side, sistema de permisos granular y portal de clientes. En
este proyecto se emplea como infraestructura completa del sistema, cubriendo
las capas de datos, lógica de negocio, presentación y seguridad.

  - Edición utilizada: Developer Edition (Sandbox)
  - URL de login: test.salesforce.com
  - Versión de API: 65.0 (componentes individuales), 59.0 (proyecto SFDX)


4.2. Apex
─────────

Apex es el lenguaje de programación propietario de Salesforce, de tipado
fuerte y sintaxis similar a Java. Se ejecuta en el servidor de Salesforce y
permite implementar lógica de negocio, controladores para componentes LWC,
triggers, pruebas unitarias y callouts a APIs externas.

En OptiSecure Solutions, Apex se utiliza para implementar 8 clases de
controlador y 4 clases de prueba unitaria.


4.3. Lightning Web Components (LWC)
────────────────────────────────────

Lightning Web Components es el framework de desarrollo de interfaces de
usuario de Salesforce, basado en estándares web (Web Components, Shadow DOM,
Custom Elements, ES Modules). A diferencia del framework Aura (su predecesor),
LWC se apoya directamente en las APIs nativas del navegador, lo que mejora
el rendimiento y la interoperabilidad.

El proyecto consta de 10 componentes LWC funcionales que implementan la
totalidad de la interfaz de usuario, tanto para el portal de clientes como
para los paneles de administración internos.


4.4. SOQL (Salesforce Object Query Language)
────────────────────────────────────────────

SOQL es el lenguaje de consultas de Salesforce, similar a SQL pero
específicamente diseñado para el modelo de objetos de la plataforma. Soporta
consultas con relaciones padre-hijo, filtros de seguridad (WITH
SECURITY_ENFORCED), y campos de fórmula.

Se emplea en todos los controladores Apex del proyecto para la lectura de
datos de cámaras, solicitudes, casos y usuarios.


4.5. Experience Cloud
─────────────────────

Experience Cloud (anteriormente Community Cloud) es la solución de Salesforce
para la creación de portales web destinados a usuarios externos (clientes,
partners, proveedores). Proporciona un sistema de registro/login, gestión de
licencias, branding personalizable y la posibilidad de exponer componentes
LWC en páginas públicas o autenticadas.

En este proyecto se utiliza la plantilla LWR (Lightning Web Runtime), que
ofrece un rendimiento superior al de las plantillas Aura clásicas, aunque con
la limitación de no soportar componentes Aura en las áreas de cabecera.


4.6. Google Gemini API
──────────────────────

Google Gemini es una familia de modelos de inteligencia artificial generativa
desarrollados por Google DeepMind. El proyecto utiliza el modelo gemini-2.0-
flash, seleccionado por los siguientes criterios:

  - Velocidad de respuesta: Optimizado para latencia baja.
  - Soporte de español: Capacidad nativa de procesamiento en castellano.
  - Coste: Gratuito dentro del tier de uso personal (hasta 15 RPM).
  - Formato de salida: Capacidad de generar JSON estructurado.

La API se invoca desde Apex mediante callouts HTTP POST al endpoint:
  https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent


4.7. Web Speech API
───────────────────

La Web Speech API es una interfaz del navegador que permite la transcripción
de voz a texto (Speech-to-Text) sin necesidad de servicios externos. En
OptiSecure Solutions se utiliza la interfaz SpeechRecognition, configurada
para reconocimiento continuo en español (es-ES), con resultados intermedios
habilitados para mostrar la transcripción en tiempo real.

Es compatible con los navegadores Chrome, Edge y Safari.


4.8. Flask (Python)
───────────────────

Flask es un microframework de desarrollo web para Python. En este proyecto
se utiliza como servidor de streaming de vídeo local, encargado de capturar
los fotogramas de la cámara conectada al equipo del usuario y emitirlos como
un flujo MJPEG (multipart/x-mixed-replace) a través de una ruta HTTP.

El servidor Flask actúa como un puente entre el hardware local (cámara USB o
integrada) y la aplicación web en la nube (Salesforce).


4.9. Ngrok
──────────

Ngrok es una herramienta de tunneling que expone servicios locales en URLs
públicas seguras (HTTPS). En el contexto de OptiSecure Solutions, Ngrok toma
la URL local del servidor Flask (por ejemplo, http://localhost:5000/video) y
genera una URL pública (por ejemplo, https://xxxx.ngrok-free.app/video)
accesible desde cualquier punto de Internet, incluido el portal de Salesforce.

Esta herramienta resuelve el problema fundamental de conectar un recurso
local (la cámara del usuario) con una aplicación cloud (Salesforce) sin
necesidad de configurar routers, firewalls ni servidores intermedios.


4.10. OpenCV
────────────

OpenCV (Open Source Computer Vision Library) es una biblioteca de visión por
computador para Python y C++. En este proyecto se emplea la interfaz
cv2.VideoCapture para acceder a la cámara del equipo y capturar fotogramas
en tiempo real, que posteriormente se codifican como JPEG y se emiten a
través del servidor Flask.


4.11. Herramientas de desarrollo
────────────────────────────────

  - Visual Studio Code + Salesforce Extension Pack: Entorno de desarrollo
    principal con soporte de autocompletado, despliegue y depuración para
    Apex y LWC.

  - Salesforce CLI (sf): Herramienta de línea de comandos para la gestión
    del proyecto SFDX, despliegue de metadatos y ejecución de scripts Apex.

  - Jest + @salesforce/sfdx-lwc-jest: Framework de pruebas unitarias para
    componentes Lightning Web Components.

  - ESLint + @lwc/eslint-plugin-lwc: Análisis estático de código con reglas
    específicas para el framework LWC.

  - Prettier: Formateador automático de código JavaScript, HTML y CSS.

  - Git: Sistema de control de versiones distribuido.

  - Python 3.x: Intérprete para el servidor de streaming Flask.

  - pip: Gestor de paquetes de Python para la instalación de Flask, OpenCV
    y dependencias.


4.12. Otros
───────────

5.1. Descripción general
────────────────────────

OptiSecure Solutions es una aplicación de gestión de seguridad doméstica y
empresarial desarrollada sobre la plataforma Salesforce. El nombre del
producto combina «Opti» (optimizar) con «Secure» (seguridad), reflejando su
propósito de optimizar la gestión de servicios de seguridad.

El sistema se dirige a empresas del sector de la seguridad que necesitan
gestionar de forma centralizada las cámaras de vigilancia de sus clientes,
tramitar las solicitudes de contratación de nuevos dispositivos, resolver
incidencias técnicas y ofrecer a cada cliente un portal web donde pueda
supervisar su instalación, solicitar servicios y comunicarse con el equipo
técnico.

La aplicación se compone de dos ámbitos de usuario diferenciados:

  - Ámbito de administración (CRM interno): Los operadores de la empresa
    de seguridad utilizan Salesforce Lightning Experience para gestionar
    el inventario de cámaras, aprobar o rechazar solicitudes de contratación,
    atender casos de soporte y supervisar la actividad del sistema.

  - Ámbito de cliente (portal web): Los clientes finales acceden a un
    sitio Experience Cloud donde visualizan el mapa interactivo de su
    vivienda con las cámaras posicionadas, solicitan la instalación de
    nuevas cámaras, reportan incidencias, se comunican con el equipo de
    soporte mediante chat y emiten comandos de voz procesados por IA.


5.2. Modelo de negocio
──────────────────────

OptiSecure Solutions se plantea como un producto SaaS (Software as a Service)
dirigido a empresas de seguridad. Las funcionalidades clave del modelo de
negocio son:

  - Gestión del inventario de cámaras de seguridad desplegadas en los
    domicilios de los clientes.

  - Autoservicio de contratación: los clientes solicitan la instalación
    de nuevas cámaras a través del portal; el administrador revisa,
    aprueba o rechaza la solicitud; si se aprueba, el cliente posiciona
    las nuevas cámaras sobre el plano de su vivienda.

  - Soporte técnico con trazabilidad: los clientes abren casos de
    soporte, reportan incidencias vinculadas a cámaras específicas y se
    comunican con el equipo técnico mediante un chat basado en
    CaseComments de Salesforce.

  - Asistente de voz con IA: los clientes pueden interactuar con el
    sistema mediante comandos de voz en español, que son procesados por
    Google Gemini para ejecutar acciones automáticas.

  - Streaming de vídeo en vivo: los clientes pueden visualizar la
    imagen en tiempo real de sus cámaras locales dentro del portal web
    de Salesforce, sin necesidad de aplicaciones externas.

El modelo multi-tenant de Salesforce garantiza el aislamiento de datos entre
clientes de forma nativa mediante perfiles y sharing rules, sin necesidad de
programar la separación de datos de forma manual.

6.2. Flujo de información entre módulos
───────────────────────────────────────

El sistema se articula en torno a los siguientes flujos de información:

Flujo 1 — Visualización del mapa de seguridad:
  Cliente → Portal Experience Cloud → LWC houseSecurityMap →
  Apex HouseMapController.getCamaras() → SOQL Camara__c →
  Respuesta JSON → Renderizado de chinchetas sobre el plano.

Flujo 2 — Contratación de cámaras:
  Cliente → LWC cameraContractRequest → Apex ContratacionCamaraController
  .crearSolicitud() → Insert Solicitud_Camara__c (estado: Pendiente) →
  Admin → LWC contractApproval → Apex .aprobarSolicitud() → Update
  estado: Aprobada → Cliente → LWC houseSecurityMap (drag & drop) →
  Apex .saveCamarasContratadas() → CameraInsertHelper (insert Camara__c
  + update solicitud a Completada).

Flujo 3 — Reporte de incidencia:
  Cliente → LWC cameraIncidentReport → Apex CameraIncidentController
  .createCameraIncident() → Insert Case (Type: Mechanical, Origin: Web).

Flujo 4 — Chat de soporte:
  Cliente → LWC caseViewer → Apex CaseViewerController
  .getCaseComments() / .addCaseComment() → SOQL/Insert CaseComment.

Flujo 5 — Asistente de voz IA:
  Cliente → LWC voiceAssistant → Web Speech API (transcripción) →
  Apex VoiceCommandController.processVoiceCommand() → HTTP POST
  Google Gemini → JSON {accion, parametros} → Ejecución DML
  (insert Case / consulta Camara__c) → Respuesta JSON al LWC.

Flujo 6 — Streaming de vídeo local:
  Cámara física → OpenCV (cv2.VideoCapture) → Flask (servidor MJPEG) →
  http://localhost:5000/video → Ngrok (túnel HTTPS) →
  https://xxxx.ngrok-free.app/video → LWC houseSecurityMap (<img src>) →
  Visualización en el portal del cliente.

Patrones de diseño aplicados:

  - MVC (Model-View-Controller): Separación entre objetos de datos (Model),
    plantillas LWC (View) y controladores Apex (Controller).

  - Wrapper Pattern: Clases internas en Apex (CaseInfo, ChatMessage,
    UserProfileData, SolicitudAprobada, CameraData) para encapsular y
    transportar datos entre el controlador y la vista.

  - Helper Pattern: CameraInsertHelper como clase auxiliar without sharing
    para realizar operaciones DML con elevación de privilegios controlada.

  - Observer Pattern: CustomEvents en LWC para la comunicación
    desacoplada entre componentes (evento voicecommand desde
    voiceAssistant hacia houseSecurityMap).

  - Optimistic UI: Actualización inmediata de la interfaz antes de
    recibir la confirmación del servidor (reposicionamiento de cámaras
    en el mapa, eliminación de chinchetas).

7.5. Arquitectura de streaming de vídeo local
──────────────────────────────────────────────

Para resolver el desafío de mostrar la imagen de una cámara de seguridad
conectada físicamente al equipo del cliente dentro del portal web de
Salesforce (alojado en la nube pública), se ha diseñado la siguiente
arquitectura de tres componentes:

┌─────────────────────────────────────────────────────────────────────┐
│                   EQUIPO LOCAL DEL CLIENTE                         │
│                                                                     │
│  ┌──────────────┐    ┌──────────────────┐    ┌──────────────────┐  │
│  │  Cámara USB  │───▶│  Python + OpenCV │───▶│  Flask Server    │  │
│  │  o integrada │    │  cv2.VideoCapture│    │  localhost:5000   │  │
│  └──────────────┘    └──────────────────┘    └────────┬─────────┘  │
│                                                       │             │
│                                          http://localhost:5000/video │
│                                                       │             │
│                                               ┌───────┴─────────┐  │
│                                               │  Ngrok Client   │  │
│                                               │  (túnel HTTPS)  │  │
│                                               └───────┬─────────┘  │
└───────────────────────────────────────────────────────┼─────────────┘
                                                        │
                                    https://xxxx.ngrok-free.app/video
                                                        │
┌───────────────────────────────────────────────────────┼─────────────┐
│                   SALESFORCE (NUBE)                    │             │
│                                                       │             │
│  ┌──────────────────────────────┐                     ▼             │
│  │  LWC houseSecurityMap       │  ◀──── <img src="https://...">    │
│  │  (Portal Experience Cloud)  │                                    │
│  └──────────────────────────────┘                                   │
└─────────────────────────────────────────────────────────────────────┘

Descripción del flujo:

  1. Python captura la cámara: El script utiliza OpenCV
     (cv2.VideoCapture) para acceder a la cámara conectada al equipo y
     leer fotogramas en tiempo real.

  2. Flask emite el streaming: El servidor Flask expone una ruta
     /video que entrega los fotogramas codificados como JPEG en un flujo
     MJPEG (multipart/x-mixed-replace). La URL local resultante es
     http://localhost:5000/video.

  3. Ngrok crea el túnel público: Ngrok toma la URL local y genera una
     URL pública segura (HTTPS), por ejemplo
     https://1a2b3c4d.ngrok-free.app/video. Esta URL es accesible desde
     cualquier dispositivo conectado a Internet.

  4. El LWC consume el streaming: En el componente houseSecurityMap de
     Salesforce, se utiliza una etiqueta <img> cuyo atributo src apunta
     a la URL de Ngrok. El navegador del cliente recibe los fotogramas
     MJPEG y los muestra como vídeo en tiempo real dentro del portal.

Esta arquitectura resuelve el problema de conectar un recurso local
(cámara del usuario) con una aplicación cloud pública (Salesforce) sin
necesidad de servidores intermedios permanentes, configuración de firewall
ni infraestructura de WebRTC.

8.4. Integración con Google Gemini
───────────────────────────────────

Configuración:
  - Remote Site Setting: Google_Gemini_API
    (https://generativelanguage.googleapis.com)
  - Custom Label: Gemini_API_Key (almacenamiento seguro de la clave API)

Flujo de la integración:
  1. El usuario dicta un comando de voz en el componente voiceAssistant.
  2. La transcripción se envía a VoiceCommandController.processVoiceCommand.
  3. El controlador construye un prompt con instrucciones contextuales
     que define las acciones posibles y los parámetros esperados.
  4. Se realiza un HTTP POST a la API de Gemini con el prompt del
     sistema y el texto del usuario.
  5. Gemini analiza la intención y devuelve un JSON estructurado.
  6. El controlador parsea la respuesta y ejecuta la operación DML
     correspondiente (crear Case, consultar cámaras).
  7. Se devuelve al LWC un JSON con el resultado de la operación.

Seguridad de la integración:
  - La API key se almacena como Custom Label, separada del código fuente.
  - Las llamadas HTTP se realizan desde Apex (server-side), sin exponer
    la clave en el navegador del cliente.
  - Remote Site Settings limita los dominios contactables desde Apex.


8.5. Integración de streaming de vídeo local
─────────────────────────────────────────────

La integración de streaming de vídeo local permite a los clientes
visualizar la imagen en tiempo real de cámaras de seguridad conectadas
físicamente a su equipo, directamente dentro del portal web de Salesforce.

Componentes de la solución:

  1. Servidor de streaming (Python + Flask + OpenCV):
     El script de Python utiliza OpenCV para acceder a la cámara del
     equipo (cv2.VideoCapture(0)) y capturar fotogramas en tiempo real.
     Flask expone una ruta /video que emite los fotogramas como flujo
     MJPEG mediante la cabecera Content-Type: multipart/x-mixed-replace.

  2. Túnel público (Ngrok):
     Ngrok se ejecuta en el equipo local y crea un túnel seguro que
     expone la URL local del servidor Flask en una URL HTTPS pública.
     El comando de ejecución es:
       ngrok http 5000

  3. Componente LWC (houseSecurityMap):
     El campo Video_URL__c de la cámara se configura con la URL de Ngrok.
     En el componente, se utiliza una etiqueta <img> o <iframe> cuyo
     atributo src apunta a dicha URL. El navegador recibe el flujo MJPEG
     y lo renderiza como vídeo en tiempo real.

Ventajas de la arquitectura:
  - No requiere servidores intermedios permanentes.
  - No requiere configuración de firewall ni router.
  - El túnel Ngrok proporciona cifrado HTTPS de extremo a extremo.
  - Es extensible: Ngrok puede sustituirse por soluciones empresariales
    de tunneling o por streaming WebRTC directo.

9.2. Patrones de testing
────────────────────────

Los tests siguen las convenciones estándar de Salesforce:

  - Test.startTest() / Test.stopTest(): Delimitadores de contexto que
    resetean los governor limits.
  - System.assert / System.assertEquals: Verificaciones de resultados.
  - Test.setMock(HttpCalloutMock.class, ...): Simulación de callouts

Patrón de datos de test:
  1. Crear Account.

  4. Ejecutar métodos del controlador.
  5. Verificar resultados con assertions.

10.1. Resumen de los logros alcanzados
──────────────────────────────────────

OptiSecure Solutions demuestra que la plataforma Salesforce permite
construir una aplicación de dominio específico completa — desde el modelo
de datos hasta el portal de clientes — aprovechando las capacidades nativas
de la plataforma para reducir significativamente el esfuerzo de desarrollo.

Los logros alcanzados en el proyecto son:

  - Portal de clientes funcional con registro, autenticación y acceso
    individualizado a los datos de cada cliente.

  - Mapa interactivo de seguridad con posicionamiento de cámaras por
    coordenadas, drag and drop, menú contextual y visor de vídeo en
    pantalla completa.

  - Flujo de contratación completo con solicitud desde el portal,
    aprobación/rechazo por el administrador y posicionamiento de las
    cámaras aprobadas por el cliente.

  - Sistema de soporte técnico con casos, chat basado en CaseComments y
    reporte de incidencias vinculadas a cámaras.

  - Asistente de voz con inteligencia artificial que procesa comandos en
    español y ejecuta acciones en Salesforce mediante Google Gemini.

  - Streaming de vídeo local que permite visualizar cámaras físicas
    conectadas al equipo del cliente dentro del portal web, mediante una
    arquitectura Flask + OpenCV + Ngrok.

  - Cobertura de pruebas unitarias automatizadas del backend con
    simulación de APIs externas.

  - Modelo de seguridad basado en perfiles, permisos y verificaciones
    programáticas server-side.

Salesforce ha permitido concentrar el esfuerzo de desarrollo en la lógica
de negocio de OptiSecure, delegando en la plataforma la gestión de la base
de datos, la autenticación, el control de acceso, el portal de clientes y
la API REST. No obstante, la plataforma impone sus propias restricciones
(governor limits en SOQL, Shadow DOM en LWC, limitaciones de la plantilla
LWR) que han requerido soluciones técnicas específicas documentadas en esta
memoria.


10.2. Limitaciones y mejoras para futuras versiones
────────────────────────────────────────────────────

Se identifican las siguientes líneas de mejora para futuras versiones:

  1. Planos por cuenta: Permitir que cada Account disponga de su propio
     plano de vivienda, en lugar de utilizar una imagen estática global.

  2. Streaming WebRTC: Sustituir la arquitectura MJPEG + Ngrok por un
     protocolo WebRTC que permita baja latencia y comunicación
     bidireccional.

  3. Grabación y almacenamiento: Implementar la grabación de segmentos
     de vídeo y su almacenamiento en Salesforce Files o en un servicio
     de almacenamiento externo (Amazon S3, Google Cloud Storage).

  4. Notificaciones push: Avisar a los clientes cuando una cámara
     detecte movimiento o se pierda la conexión del dispositivo.

  5. Dashboard analítico: Panel con métricas de uptime de cámaras,
     incidencias por zona y tiempos de respuesta de soporte.

  6. Integración IoT: Conectar sensores de movimiento, apertura de
     puertas y humo para ampliar el sistema de seguridad.

  7. Aplicación móvil: Desarrollar una app nativa con Salesforce Mobile
     SDK para acceder al mapa y a las cámaras desde dispositivos móviles.

  8. Multi-idioma: Utilizar Translation Workbench para soportar varios
     idiomas (actualmente el sistema opera exclusivamente en español).

  9. Múltiples cámaras simultáneas: Permitir la visualización de varias
     cámaras en cuadrícula (vista multiplex) dentro del portal.

  10. Detección de anomalías con IA: Aplicar modelos de visión por
      computador para detectar movimiento o intrusión en los flujos
      de vídeo de las cámaras.

11.2. Pasos de instalación

  1. Clonar el repositorio:
       git clone <url-del-repositorio>
       cd OptiSecure-Solutions

  2. Autenticarse en la org de Salesforce:
       sf org login web --alias miOrg --instance-url https://test.salesforce.com

  3. Desplegar los metadatos:
       sf project deploy start --source-dir force-app -o miOrg

  4. Asignar permisos:
     Desde Setup, asignar los Permission Sets necesarios y configurar
     los perfiles de Experience Cloud.

  5. Configurar Custom Label:

  9. Instalar dependencias del servidor de streaming:

      Descargar e instalar desde https://ngrok.com.

  11. Instalar dependencias Node.js (opcional, para tests):
        npm install

              CAPÍTULO 12. MANUAL DE USUARIO

================================================================================


12.1. Inicio de sesión en el portal
────────────────────────────────────

El cliente accede al portal de OptiSecure Solutions a través de la URL del
sitio Experience Cloud. Se presenta una página de login con dos columnas:
el formulario de autenticación a la izquierda y un banner corporativo a la
derecha. Tras introducir sus credenciales y pulsar «Iniciar sesión», el
sistema valida la autenticación y redirige al mapa de seguridad.


12.2. Página principal — Mapa de seguridad
──────────────────────────────────────────

Al iniciar sesión, el cliente visualiza el mapa interactivo de su vivienda.
El plano ocupa la mayor parte de la pantalla, con chinchetas que representan
las cámaras de seguridad instaladas. A la derecha se muestra una sidebar con
la lista de cámaras y su estado. En el footer se encuentran los botones de
acceso a los demás módulos (contratación, casos, asistente de voz,
incidencias) y la fecha/hora en tiempo real.


12.3. Gestión de cámaras
─────────────────────────

  - Ver vídeo en vivo: El cliente hace clic en una chincheta del mapa.
    Se abre un visor en pantalla completa con el stream de la cámara
    (YouTube embed o MJPEG local) y un badge «EN VIVO».

  - Renombrar cámara: En la sidebar, el cliente pulsa el icono de lápiz
    junto al nombre de una cámara, introduce el nuevo nombre y confirma.

  - Administradores: Pueden arrastrar nuevas chinchetas desde el tray
    lateral y soltarlas sobre el plano. También pueden eliminar cámaras
    mediante clic derecho → «Eliminar cámara».


12.4. Contratación de nuevas cámaras
─────────────────────────────────────

  1. El cliente pulsa «Contratar» en el footer.
  2. Se abre un modal con dos pestañas: «Nueva solicitud» e
     «Mis solicitudes».
  3. En «Nueva solicitud», indica el número de cámaras y los detalles.
  4. Al enviar, se genera una solicitud con estado «Pendiente».
  5. El administrador revisa la solicitud en el panel de aprobación y
     la aprueba o rechaza.
  6. Si la solicitud es aprobada, al cliente le aparece un tray de
     chinchetas en el mapa. Arrastra cada cámara al plano y pulsa
     «Guardar cámaras». La solicitud pasa a «Completada».


12.5. Sistema de soporte y chat
───────────────────────────────

  - Reportar incidencia: El cliente pulsa el botón de incidencias,
    selecciona la cámara afectada, describe el problema, indica la
    prioridad y envía. Se genera un caso de soporte automáticamente.

  - Consultar casos: El cliente pulsa el botón de casos para ver la
    lista de sus casos de soporte con badges de estado y prioridad.

  - Chat: Dentro del detalle de un caso, el cliente puede enviar y
    recibir mensajes. Los mensajes propios se muestran en burbujas
    azules a la derecha; los del equipo de soporte, en burbujas grises
    a la izquierda.


12.6. Asistente de voz con IA
─────────────────────────────

  1. El cliente pulsa el icono de micrófono en el footer.
  2. Se abre un modal con un micrófono central.
  3. Al pulsar el micrófono, se inicia la grabación de voz.
  4. La transcripción aparece en tiempo real en un área de texto.
  5. El cliente puede editar la transcripción si lo desea.
  6. Al pulsar «Enviar Comando», el texto se procesa con Google Gemini.
  7. Si la IA interpreta un comando válido (abrir caso, consultar
     cámaras), se ejecuta la acción y se muestra el resultado.


12.7. Streaming de vídeo en vivo
────────────────────────────────

Para visualizar una cámara local conectada al equipo:

  1. Iniciar el servidor de streaming en el equipo local:
       python stream_server.py

  2. Iniciar el túnel Ngrok:
       ngrok http 5000

  3. Copiar la URL pública generada por Ngrok (por ejemplo,
     https://1a2b3c4d.ngrok-free.app/video).

  4. Configurar el campo Video_URL__c de la cámara correspondiente
     en Salesforce con dicha URL.

  5. Al hacer clic en la cámara del mapa, el portal muestra la imagen
     en tiempo real transmitida desde el equipo local.

Capítulo 4.10 y 4.12. Tecnologías utilizadas — ampliación de la capa de visión
artificial e integración Python-Salesforce
───────────────────────────────────────────────────────────────────────────────

La evolución reciente del proyecto amplía la pila tecnológica inicialmente
descrita con la incorporación de un subsistema de visión artificial en el
borde (edge computing). A las tecnologías ya documentadas para streaming
local, concretamente Flask, OpenCV y Ngrok, se añaden dos elementos de gran
relevancia técnica: por un lado, el modelo YOLOv8 de Ultralytics, utilizado
para la detección de personas en tiempo real sobre los fotogramas capturados
por la cámara; por otro, la librería simple-salesforce, empleada para
establecer la comunicación autenticada entre el servidor Python y la org de
Salesforce.

Esta ampliación modifica de forma significativa el papel del servidor local.
Deja de ser un mero emisor de vídeo MJPEG para convertirse en un agente capaz
de analizar la escena, tomar decisiones automáticas, registrar evidencias y
sincronizar eventos con la plataforma CRM. Desde el punto de vista académico,
esta incorporación resulta especialmente relevante porque demuestra la
viabilidad de una arquitectura híbrida en la que el procesamiento intensivo de
imagen se ejecuta cerca de la fuente de datos, mientras que Salesforce asume
la trazabilidad de negocio, la persistencia documental y la interacción con el
usuario final.

  RF-14  El sistema debe detectar automáticamente la presencia de personas

  RF-15  El sistema debe generar, tras una detección válida, una

  RF-16  El sistema debe permitir al usuario consultar desde el portal las

  RF-17  El asistente de voz debe interpretar órdenes orientadas a la

─────────────────────────────────────────────────────

  RNF-07  Baja latencia operativa: El análisis de vídeo debe realizarse en el

generación del evento, evitando transferir continuamente vídeo bruto
a servicios externos para su procesamiento.

  RNF-08  Trazabilidad: Cada detección relevante debe quedar registrada como

  RNF-09  Resiliencia: El subsistema de videovigilancia debe tolerar fallos

  RNF-10  Seguridad de la integración: La comunicación entre el servidor de

Capítulo 6.2. Flujo de información entre módulos — ampliación
──────────────────────────────────────────────────────────────

La incorporación del subsistema de seguridad con IA introduce un nuevo flujo
de información transversal que conecta la capa física de captura de vídeo con
los módulos de interacción y seguimiento del portal. El flujo ampliado puede
describirse del siguiente modo:

  1. La cámara conectada al equipo local proporciona fotogramas al servidor
     Python mediante OpenCV.

  2. El proceso de análisis ejecuta inferencias con YOLOv8 sobre cada frame y
     determina si existe una detección de persona con una confianza superior
     al umbral definido.

  3. Cuando la detección es válida y no existe una alerta reciente dentro del
     periodo de enfriamiento, el sistema inicia la grabación automática de un
     clip corto de evidencia y lo conserva temporalmente en almacenamiento
     local.

  4. Una vez finalizada la grabación, el servidor genera una URL pública del
     clip a través de Ngrok y realiza una llamada HTTP al endpoint Apex REST
     responsable de registrar la alerta en Salesforce.

  5. Salesforce crea una actividad de tipo Task vinculada al contacto de la
     cámara, incluyendo en su descripción la referencia temporal del evento y
     el enlace estructurado al clip capturado.

  6. En paralelo, el propio servidor Python sube la evidencia multimedia a
     Salesforce Files y la relaciona con el registro Camara__c correspondiente,
     consolidando así la trazabilidad técnica y documental del evento.

  7. Finalmente, los componentes LWC del portal consultan estas alertas y
     evidencias para mostrarlas al usuario, mientras que el asistente de voz
     puede invocar su apertura directa mediante comandos en lenguaje natural.

Este flujo extiende el sistema desde una lógica puramente reactiva basada en
interacción del usuario hacia un modelo proactivo de supervisión, en el que
la plataforma no solo muestra información, sino que detecta eventos,
construye evidencias y genera acciones de seguimiento dentro del ecosistema
Salesforce.


Capítulo 7.1, 7.3, 7.4 y 7.5. Diseño del sistema — ampliación de la
arquitectura de vigilancia inteligente
───────────────────────────────────────────────────────────────────────────────

Desde el punto de vista del diseño, el proyecto evoluciona hacia una
arquitectura híbrida edge-cloud organizada en tres niveles claramente
diferenciados. En primer lugar, la capa de adquisición y análisis local, que
engloba la cámara física, OpenCV, el modelo YOLOv8 y la lógica de grabación de
clips. En segundo lugar, la capa de exposición segura, formada por Flask y el
túnel Ngrok, responsable de publicar tanto el streaming en vivo como los clips
generados. En tercer lugar, la capa cloud sobre Salesforce, donde residen la
lógica de integración Apex, la persistencia de notificaciones y evidencias, y
los componentes LWC consumidos desde Experience Cloud.

Esta distribución de responsabilidades permite situar el cálculo intensivo en
el borde, cerca de la fuente de datos, y reservar la plataforma Salesforce
para las funciones de negocio: identificación de propietarios, control de
acceso, consulta de histórico, trazabilidad documental y presentación de la
información al usuario. Se trata de una decisión arquitectónica especialmente
adecuada para un TFG de estas características, ya que combina el valor de una
plataforma empresarial SaaS con técnicas de visión por computador ejecutadas
fuera de la nube.

En lo relativo al módulo backend, la ampliación introduce una separación clara
entre controladores de integración y controladores de consulta. El endpoint
REST recibe eventos desde el mundo físico y los traduce a entidades de
negocio, mientras que el controlador de clips y alertas recupera y filtra la
información para los distintos perfiles de usuario. En paralelo, el módulo
frontend se enriquece con nuevos componentes especializados en consumo de
evidencias y notificaciones, manteniendo la coherencia visual y funcional del
portal original.


Capítulo 8.1. Clases Apex — lógica de negocio (ampliación)
───────────────────────────────────────────────────────────

PersonDetectionController

Se ha incorporado un endpoint Apex REST específicamente diseñado para recibir
eventos de detección procedentes del servidor Python. Su responsabilidad
principal consiste en deserializar el cuerpo JSON recibido, localizar la
cámara afectada, resolver el contacto asociado y generar una actividad de tipo
Task en Salesforce. Dicha actividad actúa como notificación persistente del
evento de seguridad y queda ligada al propietario de la cámara, lo que permite
su consulta posterior desde el portal Experience Cloud.

Una decisión de diseño destacable reside en la forma de encapsular la URL del
clip dentro de la descripción de la actividad mediante marcadores explícitos.
Esta estrategia simplifica la extracción posterior del enlace sin necesidad de
introducir un objeto personalizado adicional, preservando al mismo tiempo la
compatibilidad con la estructura estándar de Salesforce.

SecurityClipController

Junto al endpoint REST se ha implementado un controlador Apex orientado a la
consulta de evidencias multimedia y alertas de detección. Este controlador
proporciona métodos para recuperar clips asociados a una cámara concreta, al
conjunto de cámaras de un contacto o al usuario autenticado, así como para
obtener las alertas recientes derivadas de las detecciones automáticas.

La clase adopta un enfoque orientado a wrappers, devolviendo estructuras de
datos específicas para clips y notificaciones. Además, incorpora un helper
interno sin sharing para facilitar a usuarios de portal el acceso controlado a
las actividades de alerta, manteniendo la lógica de filtrado por contacto y
la separación entre usuarios internos y externos. Desde la perspectiva del
diseño software, esta aproximación favorece la reutilización del código y la
claridad contractual entre backend y frontend.

VoiceCommandController — ampliación funcional

El controlador del asistente de voz ya no se limita a crear casos o consultar
cámaras activas. La integración actual amplía el catálogo de acciones para
permitir la apertura directa de una cámara concreta, la apertura de una
notificación determinada y el recuento de detecciones registradas durante el
día. Esta evolución convierte al asistente en un mecanismo real de navegación
operativa sobre la aplicación, y no únicamente en una interfaz de entrada de
datos.


Capítulo 8.2. Componentes LWC — interfaz de usuario (ampliación)
──────────────────────────────────────────────────────────────────

clipNotifications

Se ha desarrollado un componente LWC encargado de presentar al usuario el
listado de notificaciones recientes de detección. Cada alerta muestra la
cámara afectada, la fecha y hora del evento y el contacto asociado cuando
procede. Desde la propia tarjeta es posible abrir el clip de seguridad o
eliminar la alerta del listado una vez revisada. Este componente traduce un
evento técnico de detección en un elemento de interfaz comprensible, trazable
y accionable para el usuario final.

securityClipViewer

Como complemento al visor de notificaciones, se ha añadido un componente LWC
específico para consultar clips de seguridad vinculados a las cámaras. El
usuario puede acceder a un histórico ordenado cronológicamente, reproducir el
contenido almacenado y descargar la evidencia si resulta necesario. Este
componente refuerza el papel del sistema no solo como herramienta de
monitorización en vivo, sino también como repositorio de evidencias de
seguridad.

voiceAssistant — ampliación de comportamiento

El componente de asistente de voz mantiene la captura de audio mediante Web
Speech API y la posterior interpretación semántica por parte de Gemini, pero
amplía su repertorio funcional con acciones orientadas a la navegación. En la
versión extendida, el usuario puede formular órdenes como «abre la cámara del
jardín», «abre la última notificación» o «cuántas notificaciones ha habido
hoy», obteniendo una respuesta inmediata sobre la propia interfaz. Esta
evolución mejora la accesibilidad y reduce la fricción de uso en escenarios
de supervisión rápida.


Capítulo 8.4. Integración con Google Gemini — ampliación del asistente de voz
───────────────────────────────────────────────────────────────────────────────

La integración con Google Gemini ha evolucionado hacia un esquema de
orquestación más robusto y determinista. En lugar de limitarse a interpretar
intenciones genéricas, el sistema define un prompt estructurado que obliga al
modelo a devolver un JSON con acciones cerradas y parámetros concretos. Este
contrato minimiza ambigüedades, simplifica el parseo en Apex y reduce el riesgo
de respuestas libres difíciles de explotar de forma segura.

La ampliación introduce nuevas acciones semánticas relacionadas con la capa de
videovigilancia: apertura de cámaras por nombre, zona o referencia numérica;
apertura de notificaciones por criterio temporal; y recuento de detecciones
diarias. De este modo, la inteligencia artificial deja de operar únicamente
como un parser de incidencias y pasa a funcionar como una interfaz conversacional
de control sobre los activos de seguridad y sus eventos asociados.

Conviene destacar, además, que esta integración sigue un principio de diseño
seguro: el audio no se envía directamente a Gemini. La voz del usuario se
transcribe previamente en el navegador y únicamente el texto resultante viaja
a Apex para su posterior envío al modelo. Esta separación reduce exposición de
datos, simplifica la arquitectura y aprovecha las capacidades nativas del
navegador para la fase de reconocimiento de voz.


Capítulo 8.5. Integración de streaming de vídeo local — evolución hacia un
servidor de seguridad con IA
───────────────────────────────────────────────────────────────────────────────

La implementación del servidor Python ha experimentado un salto cualitativo.
La versión actual no se limita a exponer el stream de la cámara, sino que
incorpora un ciclo completo de supervisión: captura de fotogramas, inferencia
con YOLOv8, anotación visual de detecciones, grabación automática de clips,
publicación segura de evidencias y sincronización con Salesforce.

En el plano interno, la solución mantiene un hilo de procesamiento continuo
que lee frames desde la cámara, dibuja bounding boxes sobre las detecciones y
actualiza el frame anotado que consume el streaming MJPEG. Cuando se detecta
una persona, el sistema genera un clip corto de evidencia y aplica un periodo
de enfriamiento configurable para evitar la multiplicación de alertas frente a
una misma presencia prolongada. Esta medida resulta importante tanto para la
estabilidad operativa como para la utilidad real de las notificaciones.

Otro aspecto reseñable es la preocupación por la robustez de ejecución. El
servidor controla fallos consecutivos de lectura desde la cámara e intenta la
reconexión automática del dispositivo cuando detecta pérdida de señal. Del
mismo modo, separa en hilos de fondo las tareas potencialmente más lentas,
como la notificación a Salesforce o la subida del clip, evitando bloquear el
proceso principal de captura y análisis. Desde una perspectiva de ingeniería,
esta decisión permite mantener la continuidad del streaming incluso cuando se
producen operaciones de integración con servicios externos.

Finalmente, la publicación de rutas específicas para estado, listado de clips,
stream en vivo y reproducción del clip proporciona una interfaz de servicio
clara, que facilita tanto la depuración técnica como la integración con el
portal. La solución resultante demuestra una transición madura desde un
prototipo de visualización hacia un auténtico servidor de seguridad con IA.


Capítulo 10.1 y 10.2. Conclusiones y trabajo futuro — actualización tras la
ampliación funcional
───────────────────────────────────────────────────────────────────────────────

La funcionalidad añadida confirma una de las ideas centrales del proyecto:
Salesforce puede actuar como núcleo de una solución empresarial compleja, aun
cuando parte del procesamiento especializado se ejecute fuera de la propia
plataforma. La incorporación del subsistema de detección automática de
personas, las notificaciones de seguridad y la navegación conversacional sobre
cámaras y alertas eleva el proyecto desde un portal de gestión y visualización
hacia una plataforma de supervisión inteligente con capacidad de reacción.

Asimismo, esta ampliación materializa parcialmente una de las líneas de mejora
que en la redacción original se situaban como trabajo futuro: la aplicación de
técnicas de visión por computador al dominio de la seguridad. En la versión
actual ya existe un prototipo funcional centrado en la detección de personas y
en la generación de evidencias automáticas. No obstante, siguen abiertas varias
líneas de evolución, como la calibración avanzada de falsos positivos, la
detección de múltiples clases de eventos, la normalización del almacenamiento
multimedia, la ejecución de pruebas end-to-end y la sustitución de Ngrok por
una solución de exposición segura más adecuada para producción.


Capítulo 11.1 y 11.2. Manual de instalación — dependencias y configuración
adicional para la capa de IA y notificaciones
───────────────────────────────────────────────────────────────────────────────

La ampliación funcional introduce requisitos adicionales en el entorno Python.
Además de Flask y OpenCV, es necesario disponer de las librerías de inferencia
YOLOv8 y de integración con Salesforce. En términos prácticos, el entorno debe
incluir dependencias equivalentes a ultralytics, simple-salesforce y el resto
de paquetes auxiliares requeridos por el servidor de seguridad. Igualmente,
debe asegurarse la disponibilidad del modelo ligero de YOLOv8 empleado en el
prototipo y del directorio destinado al almacenamiento temporal de clips.

La configuración del servicio exige también la definición de variables de
entorno para las credenciales de Salesforce, el dominio de autenticación, el
identificador de la cámara, el contacto asociado y la URL base de Ngrok. Esta
decisión es coherente con las buenas prácticas de despliegue, ya que separa la
configuración sensible del código fuente y facilita la adaptación del sistema a
distintos entornos de ejecución.

Desde el lado de Salesforce, la instalación debe contemplar el despliegue del
endpoint REST de detección, del controlador de clips y alertas, y de los
componentes LWC encargados de mostrar notificaciones y evidencias. Del mismo
modo, resulta necesario mantener configurado el dominio expuesto por Ngrok como
origen de confianza para el portal, así como verificar que los perfiles y
permission sets permitan el acceso controlado a las clases Apex y a los datos
derivados de las detecciones.


Capítulo 12.6 y 12.7. Manual de usuario — ampliación del uso del asistente y
de la gestión de notificaciones de seguridad
───────────────────────────────────────────────────────────────────────────────

La versión ampliada del asistente de voz permite una interacción más rica que
la documentada inicialmente. Además de reportar incidencias o consultar el
estado general del sistema, el usuario puede emplear órdenes como «abre la
cámara de la entrada», «muéstrame la cámara dos», «abre la última
notificación» o «cuántas notificaciones ha habido hoy». El sistema interpreta
la orden, ejecuta la acción correspondiente y muestra de forma directa la
cámara o el clip de seguridad solicitado, reduciendo el número de pasos
manuales necesarios para acceder a la información relevante.

En paralelo, el portal incorpora un nuevo flujo de consulta de eventos de
seguridad. Cuando el sistema detecta una persona, la notificación aparece en
el módulo correspondiente junto con la marca temporal y la referencia de la
cámara. El usuario puede abrir el clip asociado para revisar la escena o, una
vez atendido el evento, eliminar la alerta del listado. Este comportamiento
introduce una experiencia más próxima a la de una plataforma de vigilancia
operativa real, donde la detección, la revisión de evidencias y la gestión del
evento forman parte de un mismo ciclo de uso.

Por último, el concepto de «streaming en vivo» deja de entenderse únicamente
como visualización pasiva. La cámara conectada al equipo sigue pudiendo verse
en tiempo real desde el mapa, pero ahora esa misma señal alimenta un proceso
de análisis automático que puede generar eventos y evidencias sin intervención
del usuario. Esta evolución refuerza el valor del sistema como herramienta de
supervisión inteligente y no solo como visor remoto de cámaras.
